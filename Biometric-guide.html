VoltBuilder + Spring Boot Biometric MVP Guide

Reusable playbook for hybrid mobile apps (VoltBuilder/Cordova wrapper + Spring Boot backend + OAuth2/OpenID Connect provider).

Objective: Enable this user journey with minimal backend change:

  1. First sign-in uses normal OAuth login (username/password).
  2. Later launches use local biometric check + refresh token flow.
  3. If refresh succeeds, app opens without username/password prompt.
  4. If refresh fails, app falls back to normal OAuth login.

1) Generic Architecture

  • Wrapper layer (VoltBuilder/Cordova): local biometric prompt, secure storage, startup auth orchestration.
  • Backend layer (Spring Boot): token utility endpoints and session bootstrap endpoint.
  • Identity Provider: any OIDC provider supporting authorization code and refresh token grant.

Important: biometrics here are a local gate and convenience re-auth trigger. Backend trust still comes from OIDC tokens.

2) Required Mobile Plugins

  • cordova-plugin-fingerprint-aio (or equivalent biometric plugin)
  • cordova-plugin-secure-storage-echo (or equivalent secure storage plugin)

Also include platform biometric permissions in the wrapper config.

3) VoltBuilder Wrapper Folder Convention (Required)

If wrapper files do not exist yet, create them under:

src/main/resources/static/<application-name>-mobile/

Naming rule: folder must follow exactly application name-mobile (kebab-case recommended), for example:

  • hr-mobile-v1 (current project convention)
  • payroll-mobile
  • attendance-mobile

Use one wrapper folder per application to keep build artifacts isolated and repeatable.

4) Steps to Generate VoltBuilder Files (If Missing)

  1. Create wrapper root folder: 
    src/main/resources/static/<application-name>-mobile/
  2. Create minimum required files inside wrapper folder:
    • config.xml (Cordova/VoltBuilder app config)
    • index.html (bootstrap page + biometric startup script)
    • res/icon/android/icon.png (app icon)
  3. In config.xml include:
    • <content src="index.html" />
    • plugins: cordova-plugin-fingerprint-aio, cordova-plugin-secure-storage-echo
    • Android biometric permissions (USE_BIOMETRIC, USE_FINGERPRINT)
    • navigation/access rules for your backend host
  4. In index.html:
    • load cordova.js
    • on deviceready, run biometric check
    • execute refresh-first flow, then fallback to OAuth bootstrap
  5. Set BACKEND_BASE_URL to your deployed Spring Boot domain.

5) Backend Endpoints (Minimal Set)

EndpointPurposeAuth Requirement
GET /mobile/biometric/bootstrap If server session exists, redirect to app dashboard; otherwise redirect to OIDC login endpoint. Public
GET /mobile/auth/tokens Expose current authenticated session tokens so wrapper can store refresh token after first login. Authenticated session
POST /mobile/auth/refresh Accept refresh token, call IdP token endpoint, return new access/refresh tokens. Public (MVP) or protected (later)
POST /mobile/auth/session/login Accept access token, validate with IdP userinfo/introspection, create server session. Public (MVP) or protected (later)

6) Step-by-Step Implementation

1 Wrapper startup sequence

  1. On deviceready, run biometric availability check.
  2. If available, show biometric prompt.
  3. On biometric success, call attemptRefreshLogin().
  4. If biometric unavailable/fails policy, fallback to bootstrap route.

2 Implement refresh-first logic in wrapper

  1. Read refresh_token from secure storage.
  2. If missing, open /mobile/biometric/bootstrap.
  3. If present, call POST /mobile/auth/refresh.
  4. On success, save new tokens to secure storage.
  5. Call POST /mobile/auth/session/login with new access token.
  6. On success, redirect to dashboard; on failure, fallback to bootstrap route.

3 Capture tokens after first OAuth login

  1. After app session loads, call GET /mobile/auth/tokens with session credentials.
  2. Persist returned refresh_token and access_token in secure storage.

4 Backend bootstrap route

In the controller:

  • If Authentication exists and is authenticated → redirect to dashboard.
  • Else → redirect to provider-specific OAuth authorization entry route.

5 Backend refresh route

  • Receive refresh token.
  • Call OIDC provider token endpoint with grant_type=refresh_token, client_id, client_secret.
  • Return token response payload (access token, refresh token, expiry).

6 Backend session-login route

  • Receive access token.
  • Validate token using provider userinfo or introspection endpoint.
  • Create Spring SecurityContext, attach to HTTP session.
  • Return JSON with redirect target.

7 Security configuration update (minimal)

Allow only the small bootstrap/refresh/session-login routes that must be reachable before session restoration.

7) Generic Wrapper Pseudocode

onDeviceReady():
  biometricCheck()
    .then(showBiometricPrompt)
    .then(attemptRefreshLogin)
    .catch(fallbackToBootstrap)

attemptRefreshLogin():
  refreshToken = secureStorage.get("refresh_token")
  if !refreshToken: return fallbackToBootstrap()

  refreshed = POST /mobile/auth/refresh { refreshToken }
  secureStorage.set("access_token", refreshed.accessToken)
  secureStorage.set("refresh_token", refreshed.refreshToken)

  session = POST /mobile/auth/session/login { accessToken: refreshed.accessToken }
  if session.authenticated:
      redirect(session.redirectUrl)
  else:
      fallbackToBootstrap()

8) Generic Spring Boot Mapping (Where to Put Things)

  • Controller package: create a mobile auth controller (for token endpoints + bootstrap).
  • Security config: permit bootstrap/refresh/session-login routes only as needed.
  • OAuth settings: in application.yml, keep provider URLs and mobile client credentials configurable.
  • Wrapper assets: in static wrapper folder, centralize BACKEND_BASE_URL.

9) Configuration Template

Keep these as environment-driven values in every app:

  • BACKEND_BASE_URL (wrapper)
  • OIDC_ISSUER_URL / provider realm URL
  • MOBILE_CLIENT_ID
  • MOBILE_CLIENT_SECRET
  • OAuth redirect URI for mobile registration

10) Test Matrix (Reusable)

  1. First launch, no stored token → biometric → OAuth login shown.
  2. Post-login token capture successful.
  3. Second launch, valid refresh token → biometric → no credential prompt.
  4. Expired/invalid refresh token → biometric → fallback OAuth login.
  5. No biometric hardware or plugin unavailable → fallback bootstrap/OAuth path works.

11) Known MVP Limits

  • This guide is for functional MVP reuse across apps, not full production hardening.
  • Later phases can add stronger route protection, device binding, and advanced threat controls.

Result: With this pattern, all VoltBuilder + Spring Boot apps can implement consistent biometric-assisted re-login with minimal backend impact.
